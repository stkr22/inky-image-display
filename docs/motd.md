# Display jobs & the Message of the day (MOTD)

A **display job** is a pure content generator that targets a **grid** (see
[grids.md](grids.md)). It uses the same external-worker claim model as the
Immich/Gemini sync jobs: the sync worker's `display` cron claims due jobs
(`POST /api/display-jobs/claim-due`), generates the story and per-panel
screens out of process, and registers the result as an **image group**
targeting the job's grid. *When* that content appears on the panels is the
grid's business: groups enter the grid's content queue, and the grid's
daily display schedule shows the next queue entry — a fresh generated
group front-runs it naturally. New content formats — quotes, art with
explanations — become new job types that reuse the same claim, group and
queue machinery.

The **MOTD** is the first job type: a daily positive story — great science,
community achievements, positive humor, everyday heroism — generated by
Gemini, split into short content parts, one per grid slot.

A single display participates by living in a one-panel grid; there is no
separate device-targeting path.

## Content parts

One generated story decomposes into six parts, each rendered as a full
screen at every mapped panel's exact resolution:

| Part | Screen |
|------|--------|
| `what` | What happened and how it works — the new work and its concrete mechanism |
| `why` | Why it matters — the problem it solves, why it was hard, what it enables next |
| `when` | When it happened and the timeline behind it |
| `image` | AI illustration of the story (uses the GenAI prompt presets; new jobs default to the seeded `e_ink_scene` preset) |
| `qr` | QR code linking to the source article ("More details") |
| `takeaway` | A surprising or human detail behind the story |

Text parts can be combined two-per-screen with compound keys such as
`what+when` — useful when the grid has fewer panels than parts. The QR
screen is skipped automatically when the story has no source URL.

The operator UI assigns exactly **one** part (single or compound) per grid
slot. A slot given several parts becomes a multi-image slot sequence in
the group — that panel cycles through them, one per grid refresh. Slots
without a mapping leave their panel untouched.

## Story sources

Configured per job (`source_mode`):

- **`grounded`** — Gemini with Google Search grounding: a real story from
  roughly the last week, with the source URL taken from grounding metadata
  (never from model text, which fabricates URLs).
- **`knowledge`** — model knowledge only: timeless/historical stories, no
  web search, no source URL (and therefore no QR screen).

The operator-editable prompt (`content_prompt`, default in
`inky_image_display_shared.motd.DEFAULT_MOTD_PROMPT`) only steers themes and
tone; the structural output contract is appended in code so edits cannot
break parsing.

## Generation: the worker flow

Jobs schedule exactly like the sync jobs: `interval_minutes` (null =
manual only) with a `next_run_at` lease advanced at claim time, an
`is_active` pause switch, and a `run_requested_at` Run-now flag cleared by
the posted run report. Each claim records a `running` row in
`sync_job_runs` (job_type `display`) so the UI can show "in progress" vs
"waiting for worker"; the worker's report completes it.

One worker run per claimed job:

1. `POST /api/display-jobs/claim-due` hands out due jobs with their
   resolved slots (parts + panel pixel dimensions). Stale generated groups
   beyond the 7-day retention are pruned here too.
2. The worker generates the story (Gemini text, grounded or knowledge) and
   one AI illustration per orientation that shows the `image` part.
3. Text/QR parts are rendered by the API (`POST
   /api/display-jobs/render-part` — fonts and panel colour tuning stay
   server-side); the illustration is fitted via `/api/images/process`.
4. Screens are uploaded to S3 under `groups/{group_id}/…` and registered
   as ordinary `images` rows inside a new image group
   (`POST /api/image-groups`, then `/api/images/register` with `group_id`
   and the slot address).
5. The run report (`POST /api/sync-runs`, job_type `display`) records the
   outcome and clears the Run-now flag.

The worker cron is opt-in in the Helm chart (`sync.display.enabled`,
frequent schedule like the other workers) and needs the Gemini secret.

## Display: groups in the grid queue

Displaying is no longer a separate "session" system — a generated group is
queue content like any curated group or pool image (see
[grids.md](grids.md) for queue semantics). What replaces the session:

- **Scheduled daily display**: at the grid's `display_time` (weekday mask,
  IANA timezone) the grid shows its **next queue entry** and **holds** it —
  for `display_duration_seconds`, or until manual release when that is
  null. A freshly generated group is never-shown, so it naturally
  front-runs the replay of older content.
- **Manual**: `POST /api/display-jobs/{id}/display` (optionally with a
  retained `group_id`) or `POST /api/grids/{id}/display-group` shows a
  group now, with the same hold semantics.
- **Hold expiry / release** (`POST /api/grids/{id}/release`): the panels
  return to their own solo rotation, with a jittered rejoin so grids
  released together don't flash in lockstep.

Slot-addressed screens show simultaneously, exactly one image per panel —
a group is a frozen spread with no rotation inside it. Groups are retained
**7 days** per job (plus the newest and any currently-showing group);
pruning deletes the image rows and S3 objects at claim time.

## Data model

In `packages/shared/.../models/display_job.py`, `image_group.py`,
`image.py` and `grid.py` (migration 0024 replaced the former
`motd_messages`/`motd_screens` tables and session columns):

- `display_jobs` — job config (name, `job_type`, `target_grid_id`, content
  prompt/source/preset/model) + worker schedule (`is_active`,
  `interval_minutes`, `next_run_at`, `last_run_at`, `run_requested_at`).
- `display_job_slots` — the content part per grid slot (one part per slot;
  groups are frozen spreads). Slots address grid positions (`row`/`col`),
  not devices.
- `image_groups` — one row per generated run (or curated operator group):
  name (the headline), target grid, provenance `display_job_id`,
  description/source URL, queue position, last-displayed timestamp.
- `images.group_id` / `group_slot_row` / `group_slot_col` /
  `queue_position` — group membership and the panel a screen belongs to.
  Grouped images never enter solo or pool rotation.
- `grids` — the daily schedule (`display_*`) plus queue playback state
  (`current_group_id`, `hold_until`).

## API

Job configuration and worker hand-off under `/api/display-jobs`:

| Endpoint | Purpose |
|----------|---------|
| `GET /` | List jobs incl. slot mappings |
| `POST /` | Create a job (`name`, `job_type`, optional `target_grid_id`, `interval_minutes`) |
| `GET /{id}` | Single job |
| `PUT /{id}` | Partial update; `slots` replaces the whole mapping; `is_active` pauses; `clear_interval` switches to manual generation |
| `DELETE /{id}` | Delete; generated groups stay (provenance goes NULL) |
| `POST /claim-due` | Worker: claim due jobs with resolved slots (lease + running run rows + retention pruning) |
| `POST /render-part` | Worker: render one story part at one panel size (JPEG) |
| `POST /{id}/run-now` | Flag for an out-of-band worker run |
| `POST /{id}/display` | Show the job's newest group on its grid now; optional `{"group_id"}` redisplays a retained one |
| `GET /{id}/groups` | Retained generated groups with images, newest first |

Queue/display control lives on the grid routes — see [grids.md](grids.md).

## Operator UI

Display jobs live on the **Jobs** page (`/jobs?tab=display`): the tab lists
jobs like the sync jobs (schedule with a quick-edit dialog, run history
with in-progress/waiting states, Run now, pause switch) and links each to
its editor at `/display-jobs/{id}` — content prompt, source mode, image
preset, generation interval, and the target grid with one part per slot
plus a mini layout preview showing which panel gets which part. A
"Generated groups — last 7 days" list previews and redisplays retained
groups. The page header shows an overview of upcoming runs and recent
executions across all job types.

The **grid page** owns the display side: the "Up next" queue card (order,
reorder, show-now, release) and the "Daily schedule" card (24h time,
weekdays, timezone, duration or "until released"). Groups themselves are
listed and edited in the Groups overview on the **Images page**
(generated ones read-only + delete; curated ones: rename, grid, panel
assignments).

## Requirements

The worker needs `GEMINI_API_KEY` (story text + illustration; text model
per job `text_model_name`, image model from the selected prompt preset)
and the shared S3 writer credentials. Fonts for the text screens (DejaVu
Sans, bundled) and QR rendering (`qrcode`) ship with the API package,
which is why rendering stays an API endpoint.
