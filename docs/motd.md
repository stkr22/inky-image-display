# Display jobs & the Message of the day (MOTD)

A **display job** is a pure content generator that targets a **grid** (see
[grids.md](grids.md)): on its own interval it produces screens mapped onto
the grid's layout slots — nothing more. *When* that content appears on the
panels is the **grid's** decision: each grid carries its own display
schedule and session state, and at display time it shows the newest ready
content from the jobs targeting it. The two clocks are deliberately
decoupled — a job can pre-generate content (say, a week's worth on Monday)
long before the grid shows any of it. New content formats — quotes, art
with explanations — become new job types that reuse the same grid
targeting and generation machinery instead of inventing their own claim
system.

The **MOTD** is the first job type: a daily positive story — great science,
community achievements, positive humor, everyday heroism — generated by
Gemini, split into short content parts, one per grid slot.

A single display participates by living in a one-panel grid; there is no
separate device-targeting path.

## Content parts

One generated story decomposes into six parts, each rendered as a full
screen at every mapped panel's exact resolution:

| Part | Screen |
|------|--------|
| `what` | What happened and how it works — the new work and its concrete mechanism |
| `why` | Why it matters — the problem it solves, why it was hard, what it enables next |
| `when` | When it happened and the timeline behind it |
| `image` | AI illustration of the story (uses the GenAI prompt presets; new jobs default to the seeded `e_ink_scene` preset, whose composition handles scenes and abstract subjects — the operator can pick any other preset) |
| `qr` | QR code linking to the source article ("More details") |
| `takeaway` | A surprising or human detail behind the story (a setback, an unlikely origin, someone's persistence) |

The fields are prompted so each carries information the others do not — the
`what`/`why`/`when`/`takeaway` split maps to distinct facets (mechanism,
context, timeline, human angle) rather than four paraphrases of the same
point.

Text parts can be combined two-per-screen with compound keys such as
`what+when` — useful when the grid has fewer panels than parts. The QR
screen is skipped automatically when the story has no source URL.

The operator UI assigns exactly **one** part (single or compound) per grid
slot. The API still accepts an ordered multi-part list per slot — a slot
with several parts cycles through them at its device's refresh interval —
but that mode is no longer offered in the UI. Slots without a mapping leave
their panel untouched.

## Story sources

Configured per job (`source_mode`):

- **`grounded`** — Gemini with Google Search grounding: a real story from
  roughly the last week, with the source URL taken from grounding metadata
  (never from model text, which fabricates URLs). Two API calls: a grounded
  free-text research call, then a structuring call with a JSON schema
  (grounding and response schemas cannot be combined in one call).
- **`knowledge`** — model knowledge only: timeless/historical stories, no
  web search, no source URL (and therefore no QR screen).

The operator-editable prompt (`content_prompt`, default in
`inky_image_display_shared.motd.DEFAULT_MOTD_PROMPT`) only steers themes and
tone; the structural output contract is appended in code so edits cannot
break parsing.

## Generation schedule

Jobs generate on the same interval model as the Immich/Gemini sync jobs:
`interval_minutes` (null = manual only) with a `next_run_at` lease the
rotation-loop tick advances when it spawns a run (so a slow generation is
never started twice), and `last_run_at` stamped on success. 1440 = daily.
Unlike the sync jobs the run happens in-process in the API — there is no
worker cron involved. "Generate now" does the same on demand. A job only
generates when it has a target grid and mapped slots.

## Display lifecycle

Displaying is a **takeover of the grid**, using the grid's own claim
mechanism (`devices.claimed_by_grid_id`): while a session is active the
grid's pool rotation pauses and manual pool pushes refuse with 409.

- **Start**: by the grid's daily schedule (time-of-day + weekday mask in an
  IANA timezone, `display_*` columns on `grids`), manually per grid
  (`POST /api/grids/{id}/display-session` / "Show now"), or from a job page
  (`POST /api/display-jobs/{id}/display`, optionally with a retained
  `message_id`). The scheduled and grid-manual paths show the **newest
  ready message across all jobs targeting the grid** — several jobs can
  feed one grid, newest content wins. Each mapped slot's panel gets its
  first part pushed immediately; starting on a grid with a running session
  replaces that session's content.
- **Part rotation**: a slot with several parts cycles through them at its
  device's normal refresh interval, driven by the rotation loop's tick.
  Single-part slots are parked (no wasteful e-ink refreshes).
- **Slot edits**: screens are pre-rendered at generation time from the slot
  mapping as it was then, so edits can demand part × panel-size
  combinations with no screen. Those are rendered on demand — at session
  start, and immediately when slots are saved while a session shows that
  job (changed slots get their new first part pushed, added slots join,
  removed slots release their panel; unchanged slots are not refreshed).
  Text and QR parts re-render from the stored story; the illustration is
  re-fit from an existing image screen (raw AI bytes are not persisted), so
  a message generated with no `image` mapping at all cannot gain one
  without regenerating.
- **End**: after the grid's `display_duration_seconds`, or — when that is
  null — only via manual release (`POST /api/grids/{id}/release-session`).
  The handover is **jittered**: the session pushed every panel at the same
  instant, and each next refresh is scheduled relative to the previous one,
  so a lockstep release would leave everything flashing simultaneously
  every interval from then on. If the grid has an image pool, the grid
  itself resumes pool rotation at a random offset within its refresh
  interval (devices stay claimed by the grid). Without a pool, each device
  is released to solo rotation at its own random offset (capped at one
  hour). A panel may therefore keep the finished content up to that offset
  before its first rotation image.

Rendered screens are plain exact-size JPEGs under `motd/{message_id}/` in
the bucket, pushed through the ordinary MQTT `DisplayCommand` flow —
controllers need no job awareness.

Messages from the **last 7 days** are retained per job (plus, regardless of
age, any actively-shown message and the newest ready one, so a manual
display always has something to show); older rows and their S3 objects are
pruned after each generation. Any retained message can be redisplayed via
`POST /api/display-jobs/{id}/display {"message_id": …}` — each session
start stamps the message's `displayed_at`, shown in the UI history so
operators can see which stories were already on the panels.

## Data model

In `packages/shared/.../models/display_job.py`, `motd.py` and `grid.py`
(migration 0023 moved display scheduling from jobs onto grids; 0022
restructured the original 0015 MOTD schema):

- `display_jobs` — job config (name, `job_type`, `target_grid_id`, content
  prompt/source/preset/model) + generation cadence (`interval_minutes`,
  `next_run_at`, `last_run_at`). Multiple jobs can exist and several can
  target the same grid.
- `display_job_slots` — per grid-slot ordered part list + `rotation_index`
  (session state). Slots address grid positions (`row`/`col`), not devices,
  so swapping a panel in the layout keeps the mapping intact.
- `grids.display_*` — the display schedule (enabled, "HH:MM" local time,
  weekday mask, IANA timezone, duration) plus live session state
  (`active_message_id`, `active_since`, `active_expires_at`,
  `last_displayed_on`).
- `motd_messages` — generated story fields, status
  (`generating|ready|failed`), source URL/title; FK to `display_jobs`.
- `motd_screens` — one row per rendered part × panel size. Deliberately
  **not** `images` rows, so job content never leaks into FIFO rotation.

## API

Job configuration under `/api/display-jobs`:

| Endpoint | Purpose |
|----------|---------|
| `GET /` | List jobs incl. slot mappings |
| `POST /` | Create a job (`name`, `job_type`, optional `target_grid_id`, `interval_minutes`) |
| `GET /{id}` | Single job |
| `PUT /{id}` | Partial update; `slots` replaces the whole mapping; `clear_interval` switches to manual generation |
| `DELETE /{id}` | Delete; sessions showing its content are released first |
| `POST /{id}/generate` | 202 + task id (shared GenAI task registry) |
| `POST /{id}/display` | Start a session on the job's target grid; optional `{"message_id"}` body redisplays a retained message; 409 without a ready message |
| `GET /{id}/messages` | Retained messages with screens, newest first |

Session control on the grid routes:

| Endpoint | Purpose |
|----------|---------|
| `PUT /api/grids/{id}` | `display_*` fields set the schedule (see [grids.md](grids.md)) |
| `POST /api/grids/{id}/display-session` | Show the newest ready content now |
| `POST /api/grids/{id}/release-session` | End the session (jittered rejoin) |
| `GET /api/grids/{id}/display-status` | Active state + per-slot current part |

## Operator UI

Display jobs live on the **Jobs** page (`/jobs?tab=display`; old
`/genai?tab=jobs` and `?tab=motd` links redirect): the tab lists jobs like
the sync jobs (cadence, last generated, generate-now, delete) and links
each to its editor at `/display-jobs/{id}` — content prompt with
reset-to-default, source mode and image preset pickers, the generation
interval, the target grid with one part per slot, and a "Last 7 days"
story history (preview, per-story redisplay, generation progress).

The **grid page** owns the display side: a "Scheduled content" card with
the daily schedule (24h time, weekdays, timezone, duration or "show until
released"), the live session status with per-slot parts, Show now /
Release buttons, and links to the jobs targeting the grid.

## Requirements

`API_GEMINI_API_KEY` must be set — both story text and illustration use the
Gemini API (text model per job `text_model_name`, image model from the
selected prompt preset). Fonts for the text screens (DejaVu Sans, bundled)
and QR rendering (`qrcode`) ship with the API package.
